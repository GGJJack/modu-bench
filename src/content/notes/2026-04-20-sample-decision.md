## 요약

- 점수 저장 포맷 후보: SQLite · YAML · JSON
- 기준: 편집 쉬움 / 정적 배포 호환 / 스크립트 친화도
- 최종 선택: 도메인별 **JSON** 파일
- 이관 트리거: 점수 10,000건 초과 또는 동시 편집 충돌 반복 발생

## 계기

벤치마크 점수를 어디에 두어야 하는지 빨리 정해야 했음. 지금은 수십 건이지만 에이전트가 매일 쌓으면 곧 수천 건 단위로 늘어난다.

## 진행과정

### 시도 1 — SQLite
- 확장성은 좋지만 정적 배포와 궁합이 나쁨
- 빌드 타임에 덤프를 떠서 JSON 으로 변환하는 중간 스텝이 필요 → 복잡도 증가

### 시도 2 — YAML
- 사람이 읽기 가장 편함
- 하지만 스크립트로 다룰 때 타입이 흔들림 (숫자/문자열 자동 추론 문제)
- Zod 검증을 붙이면 해결되지만, 그럴 거면 JSON 과 이득 차이가 크지 않음

### 선택 — JSON
- 편집은 스크립트로만 한다는 규칙을 이미 갖고 있음 → 사람이 읽기 쉬울 필요성 낮음
- 정적 빌드에 그대로 임포트 가능
- 타입 안정성 우수

## 결과

- 도메인별 `src/data/benchmarks/*.json` 구조 확정
- 편집은 `skills/manage-benchmark/` 로만

## 교훈

- "지금 가장 좋은 것" 보다 "지금 규모에서 복잡도가 가장 낮은 것" 을 택한다
- 이관 조건을 미리 적어두면 나중에 결정 피로가 줄어든다
