# note — 작업 일지 · 노트 발행 스킬

개발 중 "지금 무엇을 하고 있고 어떤 고민을 했는지" 를 **히스토리 형태로 쌓고**,
작업이 어느 정도 매듭지어지면 그 흐름을 **정리된 한 편의 노트**로 발행한다.

## 2가지 모드

### A) 기록 모드 — "노트에 기록해줘"

사용자가 작업 중 **"노트에 기록해줘"** / **"노트에 남겨줘"** / **"이거 노트로 적어둬"** 등으로 말하면:

1. **대상 파일 결정**
   - 현재 진행 중인 주제(슬러그)를 파악한다.
     - 최근 대화 문맥에서 추출 (예: Mergify 자동 머지 복구 → `mergify-auto-merge`)
     - 모호하면 사용자에게 한 번만 묻는다: "어떤 슬러그로 쌓을까요? (예: `mergify-auto-merge`)"
   - 파일 경로: `reference/note-<slug>-YYYY-MM-DD.md`
     - 날짜는 **작업을 시작한 날짜**. 같은 주제를 며칠에 걸쳐 쌓으면 같은 파일에 계속 append.
     - 이미 파일이 있으면 그대로 append. 없으면 새로 만든다.

2. **헤더 (최초 생성 시만)**
   ```markdown
   # <주제 한글 제목>

   - slug: <slug>
   - 시작일: YYYY-MM-DD
   - 상태: in-progress

   ## 히스토리
   ```

3. **히스토리 엔트리 추가 (append)**
   매 기록마다 다음 블록을 `## 히스토리` 아래에 추가한다:
   ```markdown
   ### YYYY-MM-DD HH:MM
   - **상황**: 지금 무엇을 하고 있는지 (1~3줄)
   - **고민**: 어떤 결정/가설/의문을 다루고 있는지
   - **시도**: 방금 시도한 것(코드·명령·설계) — 있으면
   - **결과**: 시도가 어떻게 됐는지 (성공/실패/부분/보류)
   - **다음**: 바로 다음에 할 일 — 있으면
   ```
   - 시각은 시스템 시간 기준.
   - 필드 중 **해당 없음** 은 라인 자체를 생략 (빈 필드 금지).
   - 판단/감정/의심도 거르지 말고 있는 그대로 기록한다 (나중 오답지 자산).

4. **문제 상황 포착 시**
   사용자가 "이거 문제다" / "이슈다" / "막혔다" 류로 말하거나,
   히스토리 엔트리에 `결과: 실패/보류` 가 찍히면:
   - 같은 파일 안에 `## 문제 / 이슈` 섹션이 없으면 만들고,
   - 아래 포맷으로 한 줄 append:
     ```markdown
     - [ ] YYYY-MM-DD — <증상 한 줄>. 관련 히스토리: <위 엔트리의 시각>
     ```
   - 해결되면 체크박스 `[x]` 로 바꾸고 우측에 ` → 해결: <히스토리 시각>` 추가.
   - **별도 파일로 분리하지 않는다** — 같은 주제의 `reference/note-<slug>-*.md` 안에 누적.

5. **보고**
   - 어디에 몇 번째 엔트리로 쌓았는지 한 줄만 보고.
   - 예: "`reference/note-mergify-auto-merge-2026-04-22.md` 에 히스토리 #4 추가".

### B) 발행 모드 — "노트 발행해줘"

사용자가 **"노트 발행해줘"** / **"노트 정리해줘"** / **"노트로 정리해서 올려줘"** 등으로 말하면:

1. **대상 소스 파일 찾기**
   - 현재 진행 중인 슬러그의 `reference/note-<slug>-*.md` 파일(들) 을 모두 읽는다.
   - 날짜가 여러 개면 시간 순으로 이어 붙여 통합 히스토리로 본다.

2. **출력 경로**
   - `src/content/notes/YYYY-MM-DD-<slug>.md` — 날짜는 **발행일(오늘)**.

3. **분류 결정 (category)**
   4개 중 하나. 애매하면 사용자에게 확인.
   | category | 언제 쓰나 |
   |----------|----------|
   | `worklog` | 작업 중 마주친 문제·결정·회고 등 일상적 개발 기록 (기본값) |
   | `experiment` | 새 기술·도구·접근의 실험 기록 |
   | `memo` | 짧은 메모·아이디어·잡담 (구조 없이 가볍게) |
   | `announcement` | 공지·릴리스·변경 알림 (기록 모드 없이 직접 작성) |

4. **문서 구조**
   - `worklog` · `experiment` 는 아래 고정 구조를 따른다.
   - `memo` 는 `요약` + 본문 자유 (계기·진행과정 등 섹션 생략 가능).
   - `announcement` 는 `요약` + 본문 한 단락이면 충분.

   ```markdown
   ---
   title: "<짧고 구체적인 제목>"
   date: YYYY-MM-DD
   summary: "<2~3줄, 한 단락. 카드와 타임라인에 그대로 노출>"
   category: <announcement|worklog|experiment|memo>
   tags: [<키워드 3~6개>]
   status: published
   # cover: https://...  (이미지 있을 때만. 없으면 줄 삭제)
   readingTime: <정수, 분>
   ---

   ## 요약

   - 핵심 결론/사실 3~6개 **목록**
   - 읽지 않고도 전체를 파악할 수 있어야 함
   - 각 항목 한 줄

   ## 계기

   왜 이 작업을 시작했는지 1~3 단락. 외부 트리거(지표 회귀·요구·사건) 와 내부 질문을 함께.

   ## 진행과정

   히스토리의 시간 축을 유지하되 **서사**로 재구성. 아래 요소를 모두 포함:
   - **관찰**: 어떤 사실·현상·지표를 봤는가
   - **가설/시도**: 어떤 추측으로 어떤 방법을 시도했는가
   - **오답**: 틀렸다면 왜 틀렸는지 — **감추지 않는다**. 이게 문서의 자산.
   - **수정**: 어떻게 가설을 바꾸고 접근을 바꿨는가
   - **검증**: 해결됐다는 증거 (로그·커밋·수치·화면)

   흐름이 복잡하면 `### 1차 시도 / ### 2차 시도` 식 하위 헤딩 허용.
   코드·명령·diff 는 가독성 있으면 그대로 인용.

   ## 결과

   지금 상태가 어떻게 되었는지 — 커밋·PR·머지·배포 링크, 수치 변화.

   ## 교훈

   - 다음 번에도 유효할 원칙 3~5개 **목록**
   - 각 항목은 한 줄 규칙 + 필요 시 한 줄 부연
   - 프로젝트 고유 사실(저 설정이 저기 있다 등) 보다 **전이 가능한 규칙**을 선호
   ```

5. **검토 포인트 (발행 직전 자체 체크)**
   - [ ] 요약만 읽어도 무슨 일이 있었는지 파악되는가
   - [ ] 진행과정에 **오답 최소 1건** 이 남아있는가 (없다면 히스토리에서 실제로 없었는지 재확인)
   - [ ] 모든 주장에 근거(커밋 해시·로그·링크) 가 붙는가
   - [ ] 교훈이 "이번에만" 쓸 얘기가 아니라 재사용 가능한가
   - [ ] Zod 스키마 통과 (title/date/summary/category 필수, tags 배열)

6. **소스 파일 처리**
   - 발행 완료 후 `reference/note-<slug>-*.md` 의 헤더 `상태: in-progress` 를 `상태: published (→ src/content/notes/<filename>)` 로 바꾼다.
   - **삭제하지 않는다**. 원자료와 발행본은 분리 보존.

7. **보고**
   - 생성된 파일 경로 + category + 요약 목록 3줄 정도만 출력.

## 판단 가이드

### 기록 모드 vs 발행 모드 혼동 방지
- "기록", "남겨", "적어둬" → **기록 모드**
- "발행", "정리해서 올려", "노트로 올려", "글로 정리" → **발행 모드**
- 애매하면 **기록 모드** 가 기본. 발행은 되돌리기 번거로우므로 한 번 확인.

### 무엇을 기록할 만한가
- **기록할 것**: 결정의 근거, 틀린 가설, 실패한 명령, 예상 밖 출력, 시간을 잡아먹은 디버깅, 디자인 트레이드오프.
- **기록하지 않을 것**: 단순 진행 상황 ("파일 만들었음"), 사용자 지시 복창, 도구 결과 그대로 복붙.

### 길이
- 기록 엔트리: 엔트리당 5줄 내외 (상황·고민·시도·결과·다음). 늘어나면 쪼갠다.
- 발행본: 정해진 분량 없음. 다만 **요약 6항목 이상**은 과적재 신호.

## 파일 위치 요약

| 종류 | 경로 | 생성자 | 수명 |
|------|------|--------|------|
| 히스토리 (원자료) | `reference/note-<slug>-YYYY-MM-DD.md` | 기록 모드 | 영구 |
| 발행본 | `src/content/notes/YYYY-MM-DD-<slug>.md` | 발행 모드 | 영구, Zod 검증, 사이트에 노출 |

## 금지

- 원자료 (reference/) 를 발행본으로 바로 **이동/이름변경하지 말 것**. 항상 **복제 후 재구성**.
- 히스토리에 없는 사실을 발행본에 추가하지 말 것 (창작 금지).
- 오답(틀린 가설) 을 발행 시 **삭제하지 말 것**. 축약은 가능하나 존재는 남긴다.
- `src/data/journal/` 에 쓰지 말 것 — 거기는 Jules 에이전트 전용.
